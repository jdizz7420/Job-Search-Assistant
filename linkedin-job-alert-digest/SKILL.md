Search the user's Gmail for LinkedIn job alert emails from the last 24 hours using the Gmail connector (search_threads tool). Use a query like: from:jobalerts-noreply@linkedin.com newer_than:1d

For each matching thread, fetch the full message body (get_thread, FULL_CONTENT format). Each message has both a plaintextBody and an htmlBody. Parse job listings from plaintextBody: job title, company name, location, and a "View job" link (a linkedin.com/comm/jobs/view/<JOBID>/... URL — extract the numeric JOBID).

IMPORTANT — oversized get_thread results: these emails are large, and get_thread can exceed the ~190K character/token result limit. When that happens, the tool auto-saves the full result to a host-filesystem .txt file and its own error message may suggest spawning a subagent (the Agent tool) to parse that file. Do NOT do this — spawning a subagent from inside this scheduled task has caused the entire run to crash with "API Error: The socket connection was closed unexpectedly", silently failing the whole digest with no output and no ledger updates. Instead, when get_thread reports it's oversized, use the Grep tool directly on the saved .txt file path it returns, with bounded-context-window patterns (e.g. `.{200}\nView job: https://www\.linkedin\.com/comm/jobs/view/\d+` or `jobs/view/\d+`) to pull out job IDs, titles, companies, locations, and salary text. This has been proven reliable and must be the only path used for oversized Gmail results in this task — never delegate to a subagent/Agent tool call as part of this scheduled run, for any step.

Salary/compensation is NOT in plaintextBody but sometimes appears in htmlBody, near the job's link, in a pattern like "$110K-$140K / year". For each job, also check htmlBody for a salary range associated with that JOBID. If found, capture it as the Salary value; if not found there, leave it blank for now — it may get filled in by the browser-enrichment step below.

Deduplicate listings that appear multiple times within today's emails (same JOBID).

Then open the job tracker file at "~/Documents/Claude/Projects/Application Tracker/new-role-postings.md" (use Read). It has a markdown table with columns Job ID | Status | Role | Company | Location | Salary | Apply Link | Job Alert | Date Added, listing every job ever surfaced in a past digest. "Job Alert" is the saved search name — the quoted phrase in the email subject line.

Cross-reference: for every JOBID parsed from today's emails, skip it entirely from the digest if it already appears in new-role-postings.md (regardless of its Status). Only genuinely new JOBIDs go into today's digest.

BROWSER ENRICHMENT: the user has the Claude in Chrome extension, which can view LinkedIn while logged in — this bypasses the login-wall limitation that previously blocked fetching job pages directly. For each genuinely NEW JOBID (not already in new-role-postings.md), construct its canonical Apply Link as `https://www.linkedin.com/jobs/view/<JOBID>/` (not the tracking/comm URL from the email — the canonical form is cleaner and doesn't expire). Attempt to enrich it before writing the row:
1. Use mcp__Claude_in_Chrome__navigate (or tabs_context_mcp first if no tab exists) to open https://www.linkedin.com/jobs/view/<JOBID>/
2. Wait ~2-3 seconds after navigating before reading the page (e.g. mcp__Claude_in_Chrome__computer with action "wait"). LinkedIn lazy-loads the job description and salary block in after the initial page paint, and — separately — the "About the job" section often only renders into the DOM once it's been scrolled into view (it's virtualized, not just slow). So after the wait: use mcp__Claude_in_Chrome__computer with action "scroll" (direction "down", a few ticks) on the page BEFORE reading it, then wait ~1-2 more seconds. Reading immediately after navigate, or without scrolling, will capture loading-skeleton placeholders or skip the description entirely — this looks like a login wall or block but isn't. If the first read still comes back without an "About the job" section (or still shows gray skeleton placeholders), do a second pass: wait ~3 seconds, scroll up ~10 ticks, wait ~1 second, scroll down ~5 ticks, then read again — in practice the description frequently needs this up-then-down scroll cycle to actually mount, not just a single scroll-down. Only after this second pass fails should you conclude the page won't load.
3. Use mcp__Claude_in_Chrome__get_page_text to read the rendered page. IMPORTANT: the "About the job" section is frequently truncated behind a "…more" link even once it has fully rendered — get_page_text will return only the first paragraph or two followed by "… more" and the rest of the description (including any salary/compensation info further down, like a "Total Rewards" or pay range section) will be missing. Always check the page text for a "more" link near "About the job"; if present, use mcp__Claude_in_Chrome__computer to left_click it (find its on-screen coordinates via a screenshot first if needed) and then call get_page_text again to get the full expanded description before extracting anything. Do not conclude a job has "no salary listed" until you've expanded this section — salary info is often hundreds of words into the full description, not in the visible preview.
4. From the (expanded) page text, extract: a salary range if one is shown anywhere on the page (often under a "Pay Range" or "Total Rewards"/"Compensation" heading, sometimes broken out by geographic zone — if multiple zones are listed, capture the broadest range spanning all zones) and wasn't already found in the email. Also save the full page text to `~/Documents/Claude/Projects/Application Tracker/job-descriptions/<JOBID>.txt` using the Write tool — this lets the tailor-resume skill read the JD later without re-fetching from Chrome. Capture a 1-3 sentence summary for the chat digest output.
5. If the page still has no real "About the job" content after scrolling, waiting, and expanding "…more", or the navigation/extraction fails for any reason (timeout, blocked, no Chrome connection available, genuinely logged out), do NOT fail the whole digest — just skip enrichment for that job, leave Salary as "—" if still unknown, and continue. This step is best-effort; the core digest must still complete even if Chrome isn't open or isn't logged in at run time. Note: some postings are legitimately "Promoted by hirer · Responses managed off LinkedIn" and never have an "About the job" section at all (the description lives on an external site) — that's a real absence, not a loading bug, so don't keep retrying those indefinitely.
6. Do not click any links, accept any cookie/consent banners destructively, or take any action beyond reading/scrolling/expanding "…more" on the page.
